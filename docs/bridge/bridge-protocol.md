# xlflow Bridge Protocol

This document defines the planned protocol between the Go CLI and Excel automation bridge providers.

The protocol is a design contract for the .NET bridge migration. Existing PowerShell command output remains the current implementation until provider abstraction work lands.

## Transport

The initial bridge transport is command-scoped stdin/stdout JSON.

```txt
xlflow.exe
  -> starts bridge process
  -> writes one request JSON document to stdin
  -> reads one response JSON document from stdout
```

Rules:

- stdout is reserved for exactly one response JSON document.
- Logs must be emitted through `logs` in the response or through stderr.
- Fatal bridge failures should still return a structured JSON response when possible.
- The bridge process exit code is secondary; the response `status` and `error` are the primary machine contract.
- Long-running daemon, gRPC, and named-pipe transports are out of scope for the first .NET bridge migration.

## Request

```json
{
  "protocol_version": 1,
  "request_id": "01J00000000000000000000000",
  "command": "inspect",
  "timeout_ms": 60000,
  "payload": {
    "workbook_path": "C:\\dev\\Book.xlsm",
    "visible": false,
    "sheet": "Sheet1",
    "address": "A1:D20"
  }
}
```

Fields:

- `protocol_version`: Required integer. Current version is `1`.
- `request_id`: Required stable id generated by the Go CLI for correlation.
- `command`: Required xlflow command key handled by the bridge.
- `timeout_ms`: Optional command timeout in milliseconds.
- `payload`: Required command-specific object.

Command payloads should use snake_case JSON field names and should contain resolved paths when ambiguity would otherwise be possible.

## Response

```json
{
  "protocol_version": 1,
  "request_id": "01J00000000000000000000000",
  "status": "ok",
  "command": "inspect",
  "logs": [],
  "error": null,
  "bridge": {
    "name": "dotnet",
    "version": "0.1.0",
    "protocol_version": 1,
    "runtime": ".NET 8.0",
    "architecture": "x64"
  },
  "inspect": {}
}
```

Fields:

- `protocol_version`: Response protocol version.
- `request_id`: Echo of the request id.
- `status`: `ok` or `failed`.
- `command`: Echo of the command key.
- `logs`: Array of human-readable log messages.
- `error`: `null` on success, structured error object on failure.
- `bridge`: Metadata for the selected provider.
- Command-specific fields: Same top-level shape expected by the xlflow JSON envelope where practical.

The Go CLI maps bridge responses into the public xlflow envelope. Bridge providers should not rely on human stdout text.

## Error

```json
{
  "protocol_version": 1,
  "request_id": "01J00000000000000000000000",
  "status": "failed",
  "command": "run",
  "logs": [],
  "error": {
    "code": "EXCEL_COM_ERROR",
    "message": "Failed to run macro.",
    "phase": "macro.run",
    "source": "Microsoft Excel",
    "number": -2146827284,
    "h_result": "0x800A03EC",
    "details": {}
  },
  "suggestions": [
    "Run `xlflow doctor --json` to verify Excel and VBIDE access.",
    "Check that the macro name is fully qualified, for example `Module1.Main`."
  ]
}
```

Error fields:

- `code`: Stable machine-readable code.
- `message`: Human-readable summary.
- `phase`: Stable phase such as `bridge.resolve`, `workbook.open`, `vbide.import`, or `macro.run`.
- `source`: Error origin such as `xlflow`, `xlflow-excel-bridge`, `Microsoft Excel`, or `VBIDE`.
- `number`: Optional COM/VBA error number.
- `h_result`: Optional HRESULT hex string (e.g. `"0x800A03EC"`). The .NET bridge serializes this as `h_result` using `JsonNamingPolicy.SnakeCaseLower`, matching the public xlflow envelope.
- `details`: Optional structured diagnostic object.

Fatal Excel COM/RPC failures use `code = "excel_com_rpc_failure"` and include
`h_result` plus structured `details`. For `run`, details include the failing
macro, stage, Excel PID/HWND, worker PID, session mode/id, visibility/headless
mode, and whether the workbook was reused from a live session. The `.NET`
bridge treats disconnect-class HRESULTs such as `0x800706BE` as fatal.

## Version And Capabilities

The .NET bridge exposes metadata commands outside stdin request mode:

```powershell
xlflow-excel-bridge.exe --version-json
xlflow-excel-bridge.exe --capabilities-json
```

`--version-json` reports bridge name, version, protocol version, commit, runtime, and architecture.

`--capabilities-json` reports supported command keys. The Go resolver uses capabilities to decide whether `auto` can use .NET or must fallback.

## Fallback Semantics

Bridge fallback is owned by the Go resolver, not by the bridge executable.

- `dotnet`: strict mode; no implicit PowerShell fallback.
- `powershell`: force legacy bridge.
- `auto`: may fallback to PowerShell when .NET is unavailable, incompatible, or unsupported for the command.

Unsupported commands should return `BRIDGE_COMMAND_UNSUPPORTED` when `.NET` is explicitly selected.

## Protocol Compatibility

Protocol version `1` is the initial schema. Backward-compatible additions may add optional fields. Breaking changes require a new protocol version and resolver compatibility checks before command execution.
